### UI-2026-05-24-006 — home-page card cosmetic bugs (MAJOR, closed)

Three sub-issues in `components/context/home/PersonalDigest.vue`:

| Symptom | Root cause | Fix |
|---|---|---|
| Title "LUMEN-Inspired Hotfire Test Campaig" truncated mid-word, no ellipsis | The `.collection-name-clamp` already had `-webkit-line-clamp: 2`, but was a flex child (`flex-grow-1`) without `min-width: 0`, so it could not shrink below its intrinsic content width — defeating line-clamp ellipsis | Added `min-width: 0` and `text-overflow: ellipsis` on `.collection-name-clamp`; also added `:title="collection.name"` so a hover tooltip reveals the full name |
| AI Exchange description rendered raw `<p>...</p>` HTML | Description was bound with `{{ collection.description }}` (text interpolation), so embedded HTML was escaped to literal angle brackets | New `renderInlineDescription()` helper at `frontend/utils/inlineDescription.ts` — runs through `marked.parseInline()` + an inline-tag allow-list sanitizer; bound via `v-html` on the description-clamp |
| LUMEN description rendered raw `**markdown**` asterisks | Same: text-only binding never invoked the markdown renderer | Same: `renderInlineDescription()` turns `**bold**` into `<strong>bold</strong>` |

Both watched and recent card grids updated. Added `data-testid="collection-card-title"` and `data-testid="collection-card-description"` anchors for the e2e tests.

**Deviation from briefing:** the briefing suggested `text-overflow: ellipsis; white-space: nowrap;` on the title, which would have collapsed the existing 2-line layout. The flex-child `min-width: 0` fix preserves the 2-line clamp design intent AND restores the ellipsis — both behaviours instead of one. Advisor flagged this trade-off; verified visually against the after-screenshot.

**Security:** no DOMPurify in deps; followed the briefing's "prefer regex strip over adding a dep". `sanitizeInline()` allows `a/b/strong/i/em/code/br/span` only, drops all attributes except `href`/`title` on anchors, blocks `javascript:` / `data:` hrefs, and forces `target="_blank" rel="noopener noreferrer"` on links. Tested with `<script>`, `<iframe>`, and `onclick=` payloads (see `tests/unit/inlineDescription.test.ts`).

### UI-2026-05-24-007 — public-landing 404s on hero imagery (MAJOR, closed)

**Briefing inaccuracy noted:** the 404s are NOT on the unauthenticated landing page (`LandingPage.vue` doesn't reference these images — the home-page evidence file `p1-01-home-1920.json` shows only an avatar 404). The actual source is `frontend/pages/help.vue` → `HelpFrame.vue` → fetches `/docs/index.md`, which contains Jekyll-source `{{ '/assets/img/photo-aircraft.jpg' | relative_url }}` references. Evidence: `p1-05-help-1920.json`.

Images exist at `frontend/public/docs/assets/img/photo-{aircraft,satellite,solar}.jpg` — they just live under the `/docs/` mount, not the site root.

**Fix:** one regex in `frontend/utils/helpMarkdown.ts`. `rewriteDocsAssetPath()` is called from `resolveRelativeUrl()` and turns a leading `/assets/...` into `/docs/assets/...`. Idempotent. Doesn't touch the Jekyll Pages output (Jekyll's own `relative_url` already prefixes the site path correctly).

This means **no new images needed** and **no `<img>` references removed**.

### UI-2026-05-24-009 — file row `thumbnail?size=64` 404 noise (MAJOR, closed)

`ShepardFile` has **no `mimeType` field** (verified in `backend-client/src/models/ShepardFile.ts`). Inferred image-ness from the filename extension instead.

- Added `isImageFilename(name)` export to `composables/container/useFetchFileThumbnail.ts` — case-insensitive check against `png|jpg|jpeg|gif|webp|bmp|svg|avif|tif|tiff|ico|heic|heif`.
- `FileThumbnailCell.vue` now takes an optional `filename` prop; `load()` is gated on `isImageFilename(filename)`. Non-image rows render `mdi-file-outline` immediately — no network call, no 404 noise.
- `FilesTable.vue` passes `:filename="item.filename"` through.

Image rows still get the thumbnail (image-vs-file icon differentiation also added: image-typed rows that fail to load now show `mdi-image-outline` so you can tell "expected image" from "non-image file").

### UI-2026-05-24-015 — "Created at" column header wraps (MINOR, closed)

`FilesTable.vue` — added a `#[header.createdAt]` slot wrapping the column title in a `text-no-wrap` span. Targeted, doesn't affect other columns or other tables.

## What surprised me

- The "broken hero images" finding is on `/help`, not the public landing page — `LandingPage.vue` doesn't use those images at all. The briefing's prescription (ship images at `frontend/public/assets/img/` or replace with `<v-icon>`) was for the wrong page. The fix is a 4-line preprocessor change.
- `-webkit-line-clamp` silently fails inside a flex child without `min-width: 0`. The browser's intrinsic-content-size constraint wins over both the line clamp and any `overflow: hidden` / `text-overflow` you specify. Documented in the CSS comment for future-me.
- `marked.parseInline()` doesn't sanitize HTML — `<script>` passes through unchanged. Without DOMPurify, an explicit tag-strip pass is mandatory before `v-html`.
