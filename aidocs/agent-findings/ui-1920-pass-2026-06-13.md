## Headline

**At 1920 there is no horizontal overflow anywhere** — every walked page
measured `scrollWidth == clientWidth` (0 px overflow), no rogue horizontal
scrollbars, no clipped-off-screen content. The instance is overflow-clean at
1920. The 1920 problems are the *opposite* shape: **content that was tuned
fluid/centred for 4K is now either uncomfortably wide (single-column tool
pages stretching to ~1800px) or wastes huge horizontal bands (full-bleed list
rows with label-left/actions-right and ~1300px of dead space between).** The
worst single problem is the **`/me` profile's ~480px dead left gutter** plus
the **full-bleed list rows** that appear on Home, the "Referenced by" pane,
and "Shared with me".

The fold (1080 height) is mostly fine — the global search dropdown, dialogs,
and tool forms all sit inside 1080. No critical primary action was found
pushed below the fold on the pages that rendered.

## 1920-specific vs all-resolution split

**Genuinely 1920-specific (fine at 4K, wrong at 1920):**
- **`/me` ~480px dead left gutter** (the only finding that is *strictly worse*
  at 1920 than 4K — at 4K the proportions absorb the gutter; at 1920 it eats a
  quarter of the screen before content begins).

**1920-acute (a real problem at 1920, *equal-or-worse* at wider widths — so
not unique to 1920, but 1920 is where most users feel it):**
- Single-column tool pages stretching to ~1800px (sparql, form-preview) — **fixed**.
- Full-bleed list rows (Home "Shared with me", TS-container "Referenced by",
  presumably DataObject panels) with label-left/actions-right and a large dead
  middle band.
- TS-container chart-legend clip of long 5-tuple labels.

**All-resolution (would look the same at 1440/1920/4K):**
- Tools-landing + admin tile grids (fine).
- Collections-list table (fine), Collection detail (fine), global search (fine).
- Shapes-render "Fetch bindings" far-right placement.
- Vertical sparseness above the Collections search field.

**Not layout at all (pre-existing / nav):**
- DataObject-detail spinner hang (B1, plumbing — blocks the most important page).
- `/scene-graphs` 404 + missing tile (nav gap).
- BT-KVS 403 for `flo` (permission, not seeded-share).

## What I fixed in-PR (2 files, CSS-only, zero-risk)

Both are pure inline `max-width: 1200px` caps on the page-root `<v-container>`
of single-column text/form tool pages. The default Vuetify container max-width
at the `xl`/`xxl` breakpoint is ~1800px (measured live: container width = 1800px
on both pages at 1920); a 1200px cap is the well-established readable width for
prose + a code editor + a single form field. **It only narrows on viewports
wider than ~1280px — it is a no-op at 1440 and below (where the breakpoint cap
is already ≤1200), so it cannot regress 1440 and it improves both 1920 and 4K.**

| File | Change | Marker |
|---|---|---|
| `frontend/pages/semantic/sparql/index.vue` | `<v-container>` → `<v-container style="max-width: 1200px">` | `UI-1920-SPARQL-WIDTH` |
| `frontend/pages/tools/form-preview.vue` | `<v-container class="py-6">` → `+ style="max-width: 1200px"` | `UI-1920-FORM-PREVIEW-WIDTH` |

Visual re-verification requires a deploy (out of scope for this audit per the
"do NOT redeploy" instruction); the change is deterministic CSS on existing
elements and is verified by the gates (lint 0 errors, vitest 2007/2007 green,
and the change adds 0 typecheck errors — see Gates below).

Everything heavier was **filed, not fixed** — each carries 4K/1440-regression
risk or touches shared layout/component code the prompt told me to leave alone.

## Top 5 by effort × impact

1. **`UI-1920-WIDEROWS` (M effort, high impact).** The full-bleed
   label-left/actions-right row is the most-repeated 1920 pattern (Home, TS
   "Referenced by", almost certainly the DataObject reference panels). Fixing
   it once with a shared row-width token improves every list surface a
   researcher scans dozens of times a day. **Highest leverage.**
2. **`UI-1920-ME-GUTTER` (S–M, medium-high).** The single most visually broken
   page at 1920 and the only strictly-1920-specific finding. Cheap to fix
   (centre/cap the profile shell), high embarrassment-reduction.
3. **`UI-1920-TOOL-WIDTH-TOKEN` (S, medium).** Folds in the two caps I already
   shipped and makes the tool cluster consistent so the next tool page doesn't
   re-introduce the 1800px stretch. Pure CSS, low risk.
4. **`UI-1920-TS-LEGEND` (S–M, medium).** The clipped `rpm_fuel_pu…` legend is
   the only place a 1920 user loses *information* (a channel identity) to
   layout. Truncate-with-tooltip is cheap.
5. **B1 — DataObject-detail spinner hang (out of UI-1920 scope, CRITICAL).**
   Not a layout bug and already tracked from the 4K walk (C1), but it blocked
   auditing the most important page + the new ActionMenuButton at 1920. Flagging
   it again because *no* viewport audit can certify the DO detail page until
   this is fixed. Belongs to whoever owns the appId/numeric-id seam.

## What surprised me

- **Zero horizontal overflow at 1920, everywhere.** I expected to find at least
  one rogue scrollbar; the instance is genuinely overflow-clean. The 1920
  problems are all "too much room used badly," never "not enough room."
- **The widest single waste is `/me`'s left gutter, not a table.** I went in
  expecting many-column tables to be the 1920 pain; tables fit fine. The pain
  is empty space — a profile gutter and dead bands inside list rows.
- **Two tool pages had *no* width cap while their siblings had a 2400px cap.**
  The inconsistency (some pages 4K-tuned, some uncapped) is the real root cause,
  not any one page — hence the `UI-1920-TOOL-WIDTH-TOKEN` row.
