# UI-021 — `/containers/structureddata` "shows 0 rows" — close as WAI/misreport

**Verdict:** Misreport. The path is a Nuxt **404 — Page not found**, not an empty list.
Not a permission scope bug, not a backend query bug, not specific to alice. There
has never been a list-page route at `/containers/<type>` (without an id).

## What I found

### Reproduction (Playwright, as alice via OIDC)

URL → page body (live `https://shepard.nuclide.systems`, 2026-05-24):

| URL | Result |
|---|---|
| `/containers` | List page renders, alice sees the `lumen-inspired-runlogs` SD container |
| `/containers?selectedFilter=STRUCTUREDDATA` | List page renders, **filtered to SD only** — alice sees `lumen-inspired-runlogs` |
| `/containers/structureddata` | **`404 — Page not found: /containers/structureddata`** |
| `/containers/timeseries` | **404 — Page not found** |
| `/containers/files` | **404 — Page not found** |
| `/containers/spatialdata` | **404 — Page not found** |
| `/containers/structureddata/65` | Renders SD container detail (`65 = lumen-inspired-runlogs`) |

Console error on the 404 routes is the canonical Nuxt one:

```
[nuxt] error caught during app initialization $c: Page not found: /containers/structureddata
```

### Route layout (why)

The frontend has **one** containers list page (`frontend/pages/containers/index.vue`),
which renders `<ContainerListPage />` and is filtered by the `selectedFilter` query
parameter via `ContainerTypeSelect.vue` + `useContainerListQueryParams.ts`. The
per-type subfolders only carry **detail** pages:

```
frontend/pages/containers/
├── index.vue                              # the only list page; query-param filtered
├── files/[containerId]/index.vue          # detail
├── structureddata/[containerId]/index.vue # detail
├── timeseries/[containerId]/index.vue     # detail
└── spatialdata/[containerId]/index.vue    # detail
```

No file matches `/containers/<type>` without a `[containerId]` — Nuxt has nothing to
mount and renders its built-in 404 page (containing the literal string "0" inside
"404" plus a sparse layout). The UI-016/017 reporter quite reasonably misread that
as "list page shows 0 rows".

### Backend is fine

`POST /shepard/api/search/containers?page=0&size=20` with
`queryType=STRUCTUREDDATA` and the frontend's default empty-but-valid query
returns alice's 1 SD container correctly:

```bash
curl -H "Authorization: Bearer $ALICE" -X POST \
     -H "Content-Type: application/json" \
     https://shepard-api.nuclide.systems/shepard/api/search/containers?page=0&size=20 \
     --data '{"searchParams":{"query":"<empty-OR>","queryType":"STRUCTUREDDATA"}}'
# → {"results":[{"id":65,"name":"lumen-inspired-runlogs",…}],"totalResults":1}
```

`GET /shepard/api/structuredDataContainers` (the un-paginated v1 list) also returns
alice's 1 SD container directly. The per-id detail path that "works" in the
original report is the same one accessible from the real list page; alice's
permissions match end to end.

## Why the original report happened

The bracket UI-016 + UI-017 agent visited `https://shepard.nuclide.systems/containers/structureddata/1`
to exercise the "Referenced by" panel. When clicking around the container detail it
likely backed up one path segment (browser back, or address-bar truncation) and
saw the 404 page, then summarised it as the SD list-page being empty. The shape of
the 404 ("Page not found: /containers/structureddata" + "Go back home") is sparse
enough to be mistaken for an empty-list state.

## What surprised me

Two things:

- The frontend has **no typed list-page routes at all**, yet `WatchedContainersPanel.vue`
  uses `/containers/<type>/` as path prefixes when constructing per-id detail links —
  so the "bare typed list URL" looks plausible by analogy. A researcher used to
  REST conventions (`/api/x` listing, `/api/x/123` detail) would guess the
  list URL and hit a 404 with no signposting back to the real list.
- The Nuxt 404 page renders inside the header chrome (with nav still visible), so
  it doesn't *feel* like a hard 404. That's exactly why a finding agent skimming
  results would label it "list page shows 0 rows" rather than "page not found".
