# UI-016 + UI-017 cosmetic polish — 2026-05-24

Closes two MINOR findings from the UI Scrutinizer's 2026-05-24 sweep:
**UI-016** (SD container "Referenced by" chip density) and **UI-017**
(DataObject description edit affordance unclear). One PR, frontend-only.

## What I built

### UI-016 — SD container "Referenced by" chip cap + expand affordance

- **`frontend/components/context/display-components/semantic-annotations/SemanticAnnotationList.vue`**
  — pre-existing `limit?: number` prop now drives a default-collapsed view.
  When the caller passes `:limit="N"` and the underlying DataObject has
  more than N annotations, the surplus is hidden behind a clickable
  `+N more` chip (`data-testid="annotations-overflow-chip"`). Clicking
  expands inline; a `Show less` chip (`annotations-collapse-chip`) collapses
  it back. Both chips carry `cursor: pointer` + hover underline. Other
  callers of `SemanticAnnotationList` are unaffected — they pass no
  `limit` and continue to render all chips.

- **`frontend/components/container/LinkedDataObjectRow.vue`** — the
  "Referenced by" rows on SD / TS / File container detail pages now pass
  `:limit="3"` into `SemanticAnnotationList`. The cap matches the
  pre-existing chip-shrink styling (`font-size: 0.7rem`, narrow gap) so a
  high-density row scans as three labels + a clear overflow handle instead
  of a wall of repeating tags.

The chosen fix is option (b) from the brief — chip cap with expander.
Option (a) (cross-row "common annotations" pill bar) was deferred per
the brief's recommendation: it surfaces a design conversation about
"common-tag" semantics that's bigger than a 2-hour polish PR.

### UI-017 — DO description Edit visual cue

- **`frontend/pages/collections/[collectionId]/dataobjects/[dataObjectId]/index.vue`**
  — the description `<section>` gets a conditional `description-editing`
  class when `descEditActive === true`. The class wires:
  - `outline: 2px solid rgba(var(--v-theme-primary), 0.4)` — visible
    primary-tinted frame around the section being edited.
  - `background-color: rgba(var(--v-theme-primary), 0.03)` — subtle
    tint so the section reads as "active" without yelling.
  - A "Editing description" caption label with `mdi-pencil-outline`
    icon next to the section header, gated on
    `descEditActive`, carrying `data-testid="description-editing-label"`.
  - Two `data-testid`s on the section itself
    (`description-static` / `description-editing`) for the e2e to assert.
  - `aria-label="Edit description"` on the engage button for the e2e
    to address the right button unambiguously.

The whole-section editor pattern from BUG #139 (descEditActive,
descDraft, descSaving) is untouched — this PR only adds the visual cue,
no logic change.

## Anything deferred

- **(a) — cross-row "common annotations" pill bar** is not in this PR.
  Per the task brief, ship (b) only and let (a) surface as a separate
  design conversation. A follow-up backlog row (`UI-016a`) can capture
  it if user demand emerges.
- **Unit test for the expand/collapse state machine** was not added —
  the state is two `ref`s and three template chips, all template-only,
  no extracted helper. The e2e covers the user-observable behaviour. If
  a refactor extracts the limit / expand logic into a composable
  (e.g. `useChipOverflow()`), that composable should ship with a Vitest
  test at that point.
- **Pre-existing broken middleware blocked the build pre-redeploy** —
  resolved out-of-band during this session by commit `e3bba184`
  ("fix(frontend): useV1DeprecationMiddleware post() async — satisfies
  Middleware interface"). The original HEAD declared
  `useV1DeprecationMiddleware.ts:post()` as
  `Promise<Response | void> | void`, which fails Nuxt's tsc gate against
  the SDK's `Middleware.post: (ctx) => Promise<Response | void>`. The
  fix landed on `main` while this PR was in flight; nothing further is
  needed.
- **`feedback_capture_api_ui_annoyances.md` adds** noted but not blocking:
  - "SD container list page at `/containers/structureddata` shows 0 rows
    for the demo user — but probing direct ids 1..30 returns full pages.
    Either the list endpoint is filtered to something stricter than detail,
    or the list-page UI silently swallows results." Recommend adding to
    `aidocs/agent-findings/ui-annoyances.md` if not already present.
