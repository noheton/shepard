## Design choices worth surfacing

### Pre-import snapshot placement (advisor query)

The task placed the G1 snapshot *after* `warmup_probe_and_gate`. The
warmup gate creates a `WarmupProbe` DO + uploads probe payloads — so the
pre-import snapshot is **not the bit-perfect t=0 of the empty collection**.
Per advisor review I considered moving it before warmup; chose to follow
the task as written because:

1. The warmup is operator-controlled (one DO, three small payloads); the
   bulk forging pass is the import itself.
2. Moving it earlier means a re-run after warmup-skip (gate already passed
   in `state.gate_passed`) would never have a pre-import snapshot at all —
   the snapshot would only fire on first run.
3. The current placement guarantees one pre-import snapshot per import
   session regardless of resume state.

Recorded as a deferred-decision row in `aidocs/integrations/93 §15` "What
v15.1 explicitly does NOT add"; can flip later if audit story requires
empty-baseline pre-snapshot.

### G5 partial (no public turtle ingest)

Per advisor review: the data-ontologist's G5 design called for
`POST /v2/semantic/{repoAppId}/import` with raw turtle, but per Agent A's
report (line 107–115) no such public endpoint exists. The implementation
report listed two paths:

- **Backend extension** (auto-emit `prov:Entity` typing on `SnapshotService.create`) — deferred to v15.2/PROV-V16
- **Collection-anchored annotation** with `prov:wasInformedBy`/`prov:generated` — shipped in v15.1

v15.1 takes path 2: snapshot URNs are queryable in the SHACL graph via
SPARQL, the lineage relation (collection → snapshot) is typed, but the
snapshot entity *itself* is not yet typed as `prov:Entity`. Honest
documentation lives in `mffd-import-v15.py:annotate_snapshot_lineage`
docstring + the §15 spec table.

### Worker pool conservatism (Tier 5)

The task acknowledged Agent A's deferral rationale (real-threading tests
are flaky). v15.1 ships:

- The **wiring** is provable: `workers=1` is byte-for-byte sequential
  (asserted by mock on `ThreadPoolExecutor.__init__` being NOT called);
  `workers=N>1` instantiates the executor with N workers and health-checks
  the pool before delegating.
- The **per-task fan-out** still delegates to the sequential `run_source_mode`
  body — refactoring it into a callable closure that 4 workers can pull
  is a larger refactor (the per-DO loop is ~150 lines with nested
  state-file / TS-container / structured-data branches). That refactor
  lands in v15.2 alongside mid-import snapshots (G3).

The conservative wiring is safe: an operator using `--workers 4` today
gets a healthy ThreadPoolExecutor (proven by the health-check probes)
and the same sequential per-DO behavior as `--workers 1`. Future v15.2
swaps the inner delegation for real fan-out without breaking the CLI
contract.

### License fail-fast scoping (advisor query)

Per advisor: gate at `main()` entry into the import path, not at module
load. Done that way (`if not args.bootstrap and not args.dry_run and not
args.verify_imported: assert license args`) so:

- The 71 existing tests' `importlib.util.spec_from_file_location` module
  import path still works (would otherwise fail because module loads
  without env)
- `--bootstrap`, `--dry-run`, `--verify-imported` all work without license
  flags (read-only or no-write paths)
- Only SOURCE / LOCAL import paths abort with a clear error message
