## Regressions vs intended state

### MAJOR

**M1 — Microsections collection landing broken via `/collections/{appId}` path** (steps 3, 4, 9).
The microsections seed (`f5775aa7e` 2026-05-29) created a Collection with
appId `019e7243-f995-7914-be80-53e367aa5172` and numeric Neo4j id (apparently
not 19 either — the error message references id 19 which suggests a stale
treeview cache referencing a deleted prior seed). The frontend route
resolution at `/collections/{appId}` posts an "id" path param to the v1
endpoint which 404s; the Vuetify error toast surfaces the raw backend
message. **Two distinct error toasts:** `fetching collection roles` (step 3)
and `fetchTreeviewItem` (step 4). Both reference "id 19".
Citation invalidated: `f5775aa7e feat(examples): microsections seed` did not
verify post-seed UI rendering. The collection EXISTS in the data (it shows
on the collections list) but it is unreachable via the routed detail page.

**M2 — Frontend route does not yet accept UUID `appId` as path param**
(root cause of M1). Per CLAUDE.md L2d/`aidocs/platform/25`, the v2 shelf
introduces `appId` as native identifier. The collections-list href is using
appId, but the page-level loader is still calling v1 numeric-id endpoints.
This is the L2d work in flight (per L2 chain in CLAUDE.md API-version
policy); but the gap between the list-href and the detail-page-loader is
the immediate regression.

**M3 — TS-AXIS spatial-role chips not visible on the MFFD TS container
page** (step 5). The TS-AXIS-VERIFY recovery commit `cce854736` (and the
prelude-widening `6f28e7796`) shipped the annotation writes. The Semantic
Annotations table on `/containers/timeseries/1772` renders 8 channel rows
with measurement/device/location/symbolic-name/field — but no rendered
`spatial:axis` chips on AFP-S1 or LBR rows. Either (a) the annotations live
on AnnotatableTimeseries entities the container view doesn't query, or
(b) the chip rendering is on the TimeseriesReference detail page and not
the container channel table. Needs a follow-up open of a TS-Ref detail.
**This does not necessarily invalidate `cce854736`** — the annotation
write may have succeeded but the read path on this view doesn't surface
it. Worth a Cypher check before filing as bug.

### MINOR

**m1 — 4K vertical whitespace** (step 2). Collections list table renders
3 rows then ~1500 px of pure whitespace before footer. No skeleton, no
empty-state suggestions, no "create another collection" CTA in the void.
Pattern repeats on other table pages. Not a regression — predates
2026-05-29 — but it's painful on flo's hardware. UX-PATTERN-D (count
badges) addressed counts; the table-overflow void is a sibling concern.

**m2 — No Trace3D affordance on TS container page** (step 6). Per
`project_trace3d_view` memory + `aidocs/platform/98`, Trace3D is a
TimeseriesReference-level view. The container page is one click upstream;
adding "Visualize all in 3D" at container level is a NICE for later, not
a regression.

**m3 — Auth needs flakiness retry on cold session** (step 1). The
`loginAs()` tolerant helper succeeded on retry #1 but timed out on the
first attempt. Pattern consistent across many spec runs — not a 2026-05-29
regression. Worth a dedicated investigation when next on the auth seam.

## Top 3 issues to file

### UX-WALK-2026-05-29-01 — Microsections collection unreachable via `/collections/{appId}` (CRITICAL)
- **Severity:** CRITICAL — primary content artefact shipped today is
  invisible to users via the canonical browse path.
- **Repro:** Sign in as flo. Click "Microsections — PH2940 Composite
  Cross-Sections" on `/collections`. Observe "Not found" page + toast
  "ID ERROR - Collection with id 19 is null or deleted".
- **Root cause hypothesis:** `/collections/{id}` route loader is
  calling a v1 numeric-id endpoint with the appId UUID as the path
  param, the backend partial-parses it (or treats it as 19 from cache)
  and 404s.
- **Backlog row:** see `aidocs/16` `## UX-WALK-2026-05-29` section.

### UX-WALK-2026-05-29-02 — TS-AXIS spatial:axis chips not visible on container view (MAJOR)
- **Severity:** MAJOR — the TS-AXIS-VERIFY annotation campaign
  (`cce854736` + `6f28e7796`) was the headline TS-AXIS work today;
  visual confirmation does not surface them where users land.
- **Repro:** `/containers/timeseries/1772`. Inspect Semantic
  Annotations table rows for AFP-S1 / LBR channels. No `spatial:axis x`
  or `spatial-role: axis-x` chip column.
- **Decision needed:** are the annotations on `:AnnotatableTimeseries`
  bridge nodes and unseen by the container view query? Or is the chip
  render only on the TimeseriesReference detail page (one click
  downstream)? A 5-line Cypher probe will disambiguate before this is
  filed as a UI bug vs. a data-write bug.
- **Backlog row:** see `aidocs/16` `## UX-WALK-2026-05-29` section.

### UX-WALK-2026-05-29-03 — Frontend route should accept appId UUID (MAJOR)
- **Severity:** MAJOR — root cause of M1 / UX-WALK-...-01 and any
  future feature that links via appId rather than numeric id.
- **Repro:** Any `/collections/{uuid-v7}/...` URL surfaces backend
  "id null or deleted" toasts.
- **Resolution:** the L2d shelf (`aidocs/platform/25`) is the design
  ledger. Either accelerate the L2d resolver or add a v1 path resolver
  that walks `HasAppId.appId → numericId` before calling the v1
  endpoint.
- **Backlog row:** see `aidocs/16` `## UX-WALK-2026-05-29` section.
