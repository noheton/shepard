### What the DataObjects represent

The hierarchy in the source-side `MFFD_Demonstrator_Recursive_Tree.json` and the process
types observed in the log (`Ply_1` ‚Ä¶ `Ply_120+` under `Layup`) make this clear:

> Each `Track_NN__Run_NN_` is **one AFP tape track** on **one ply** of the upper-fuselage
> layup. 120+ plies √ó multiple tracks per ply ‚Üí 8,251 timeseries.

The natural parent hierarchy is **Collection(tapelaying) ‚Üí Ply(N) ‚Üí Track(N)/Run(N)**.

### Per-channel sketch

192 channels per track ‚âà TPS sensor data: TCP temp (multi-zone), consolidation force,
roller pressure, head position, ply count, IR pyrometer trace, robot joint states.
Channel detail confirmed only when we read a sample `Timeseries.csv` post-extract.

## 3. Bucket 2 ‚Äî `bridgewelding/` (the rework-loop substrate)

Already extracted on disk (3.7 GB). The manifest reveals **13 AF (attached-frame) parts**
named `AF_1 ‚Ä¶ AF_13` + dated `Execution_2023-04-14_HH_MM_SS` snapshots. Each `AF_N` carries
**~20 `StepMetaProcessStep` StructuredDataReferences** pointing to the **same** physical
file (`bridgewelding/AF_N/structured/StepMetaProcessStep.json`) ‚Äî a v5 export quirk where
multiple refs share one payload.

Sample StepMeta payload:
```json
[{"name":"StepMetaProcessStep",
  "data":{"_id":{"$oid":"6463357ac0f6895b3243015e"},
          "nextParameterNumber":9,
          "_meta":{"createdAt":{"$date":"2023-05-16T07:49:14.339Z"},"name":"StepMetaProcessStep"}}}]
```

This is **MongoDB-shaped JSON** with `$oid` / `$date`. The `nextParameterNumber` field
suggests this AF was advanced through 9 process parameter variants ‚Äî the rework loop the
**Industrial Quality Engineer agent** (Role 4) has been asking for. **Bridge welding ties
the AFP layup output to the joining step that comes next** in the MFFD process chain.

## 9. Posture for the import plan

The cheapest, most legible ingestion order is:

1. **Bucket 1 (mffd.tar.gz / tapelaying)** ‚Äî already in Shepard idiom; ship the demonstrator first.
2. **Bucket 2 (bridgewelding)** ‚Äî closes the AFP-‚Üí-joining bridge; tiny (3.7 GB); enables Role-4 rework story.
3. **Bucket 4 / RoboDK** ‚Äî 12 MB; immediately upgrades the physical-context narrative across every TR.
4. **Bucket 4 / thermography.7z** ‚Äî 4 GB; supplies the NDT step.
5. **Bucket 3 (.svdx + videos + TIFF)** ‚Äî needs **parser + scale work**; do in waves with feature gates.
6. **`later/`** ‚Äî defer.

The detailed plan, including idempotency contract, dry-run gates, mapping templates and
ETA, is in **`aidocs/integrations/113-mffd-real-data-import-plan.md`**.

The user-facing gaps these buckets expose are catalogued in
**`aidocs/agent-findings/mffd-feature-gaps-2026-06-02.md`**.
