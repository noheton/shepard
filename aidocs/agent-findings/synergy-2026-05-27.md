# Synergy findings — 2026-05-27

## What I found

**FAIR8 landed a scoring rubric with a deliberately hollow input.** The
`keywordCount` field in `MetadataCompletenessInputs`
(`/opt/shepard/frontend/utils/metadataCompleteness.ts`) is hard-wired to
`0` in `MetadataCompletenessCard.vue` because the query endpoint does not
exist yet. Every Collection that has keyword-style semantic annotations
scores 0 on the FsF-F2-01M (keywords) check — 5 pts left on the table for
every real dataset. The `aidocs/agent-findings/fair8-fuji-alignment-2026-05-27.md`
documents this as a tracked gap, but there is no backlog row yet with a
concrete endpoint shape.

**The AAS plugin now has a clean enable gate and zero-unlock cost.** AAS1l
shipped yesterday (`plugins/aas/src/main/java/de/dlr/shepard/plugins/aas/entities/AasConfig.java`)
adding a `:AasConfig` singleton + `GET/PATCH /v2/admin/aas/config`. The
enforcement in `AasShellsRest.aasDisabled()` checks `aasConfigService.current().isEnabled()`
and returns 501 with a clear message pointing at the config PATCH endpoint.
The next conformance slice (AAS1c — SubmodelElement drill-down) is the only
thing standing between the current surface and a usable IDTA Part-1 claim.
Nothing else in AAS1c's dependency chain is unshipped: AAS1b (single Shell),
AAS1b/submodels, and AAS1d (IDTA templates) are all done.

**The shapes/render playground surfaced a latent 5-tuple dependency it
can't yet resolve.** The playground comment (`/opt/shepard/frontend/pages/shapes/render.vue`
line 14) explicitly says "user supplies the TS container ID to complete the
pipeline until TPL2c ships". Today's commits added time-bracket display and
brush-range controls — the playground is now demonstrably useful for manual
exploration, but cannot auto-resolve channel bindings from a DataObject
context. The backlog row `VIEWS-AS-SHAPES-TRACE3D-TIMING` has already
committed this to the M1-VIEWS-AS-SHAPES-WAVE bundle (task #58 TS-IDc
migration first, then `POST /v2/shapes/render`, then Trace3D VIEW_RECIPE).

**The home page has "Shared with me" and Watched sections but no live data
signal.** `PersonalDigest.vue` shows Collections as gallery cards and a
list of shared Collections, but every card is static — name, description,
update timestamp. A researcher watching an active test campaign sees the
same card state whether data is flowing or not. The STC home-showcase
collector restarted today; the data is flowing, but that fact is invisible
from the home page.

**The AAS template surface creates a plugin-to-template derivation
opportunity that is not yet designed.** `AasShellMappingService.toSubmodelRefs`
emits top-level DataObjects as ExternalReference Submodel refs. The three
IDTA templates (Nameplate, Technical Data, Time Series Data) are seeded as
`ShepardTemplate` rows. But each payload-kind plugin (HDF5, Git, Spatial,
Video) has a natural IDTA submodel template shape — and there is no
mechanism to auto-derive one from a `PluginManifest`. Templates are created
manually today.

## Opportunities

### O1 — Keyword annotation count endpoint (5-pt FAIR score unlock)

A minimal `GET /v2/collections/{appId}/annotation-summary` (or a predicate
counts endpoint on the existing `GET /v2/semantic-annotations` surface)
would let `MetadataCompletenessCard.vue` pass a real `keywordCount` instead
of 0. The N1e `SemanticTermSearchRest` already walks n10s `:Resource` nodes;
the underlying machinery for counting annotations by predicate pattern exists
in `SemanticAnnotationDAO`. This is the cheapest possible move to improve
live dataset FAIR scores without landing any new ontology work. Concretely:
the LUMEN and MFFD Collections that already have semantic annotations would
gain 5 pts immediately.

**Shape:** `GET /v2/collections/{appId}/annotation-summary` returning
`{ "totalAnnotations": N, "annotationsByPredicate": { "skos:prefLabel": K, ... } }`.
Alternatively, a query param on the existing annotation list endpoint:
`GET /v2/semantic-annotations?subjectAppId={appId}&summarise=predicate`.

### O2 — AAS1c unblocks the first conformant IDTA Part-1 claim

With AAS1a/1b/1d/1l shipped and the 501-gate in place, AAS1c
(`GET /v2/aas/submodels/{submodelId}` + nested SubmodelElement drill-down)
is the last slice before shepard can advertise a credible IDTA Part-1
read-only profile. The `aidocs/integrations/52-aas-backend-integration.md`
§9 estimates AAS1c at M-size (~3 engineering days). The conformance
test runner (`AAS1h`) is also M-size and can be set to advisory-only
initially. This would make shepard the first open-source RDM platform
with a working IDTA AAS v3 adapter, which is a concrete differentiator
for the MFFD digital thread funding story.

**Dependency graph is clear:** AAS1c → AAS1e (write side) → AAS1f (AASX
export). Nothing in between needs new infrastructure; the DataObject
OGM model + permission checks are all in place.

### O3 — CollectionLivePreview card widget (STC demo visibility)

`CollectionGalleryCard.vue` is now used in three places: the gallery page,
the Watched section, and (as a list row) the "Shared with me" section in
`PersonalDigest.vue`. Adding a `<CollectionLivePreview>` slot to the card
would show a sparkline or "N points in last 5 min" chip for Collections that
have at least one TimeseriesReference. The backend already has
`GET /v2/timeseries-containers/{id}/channels/summary`-style endpoints. The
STC home-showcase-collector is now running; this would make the home page
reflect live data state for the first time. An acceptance test: the STC
Collection card shows a non-zero reading seconds after the collector ticks.

The risk to contain: don't trigger a TS query per card on page load — the
home page can load 30 collections and firing 30 TS queries would be a
regression. The right shape is a single `POST /v2/timeseries/batch-latest`
that accepts a list of container IDs and returns the most recent timestamp +
point count per ID in one round trip. That endpoint does not exist yet; its
absence is a design constraint to note.

### O4 — Brush-range-to-DashboardReference draft (AI1e prep, today-sized)

The render playground now captures `lastRenderStartNs` / `lastRenderEndNs`
and shows a time-bracket label. A "Save as draft dashboard" button that
serialises the current `(templateAppId, focusShepardId, brushSlider, colormapName)`
into a `DashboardReference` stub (or a structured-data JSON payload) would:
(a) persist the user's exploration session without requiring any new entity
shape, (b) feed directly into AI1e snap-dashboards when that slice ships, and
(c) decouple the Trace3D playground usability from TPL2c (which is gated on
TS-IDc). The save target can be a StructuredDataReference until
`DashboardReference` gets its own entity.

### O5 — PluginManifest → AAS Submodel Template auto-derivation

Each `PluginManifest` knows its `name()`, `description()`, and its payload
kind's entity types via `PayloadKind.entityPackages()`. A lightweight
`AasSubmodelTemplateDeriver` (run once on `StartupEvent`, best-effort like
`OntologySeedService`) could generate a `templateKind=AAS_SUBMODEL_TEMPLATE`
row for each payload-kind plugin that doesn't already have one. The template
body would be a minimal IDTA SMEC (SubmodelElementCollection) with one
property per payload kind's primary metadata fields. This would mean that
enabling a new payload plugin automatically adds a corresponding IDTA template
to `.well-known/aas-server.supportedSubmodelTemplates` — closing the loop
between the plugin registry and the AAS discovery surface without any
manual admin step.

## Surprising connections

**FAIR8 replaced `heroImage` with `keywords` on the same day hero images for
LUMEN + MFFD were added via Garage S3.** These are coincident commits
(`feat(showcase): hero images for LUMEN + MFFD via Garage S3 + Caddy /static`
and `feat(ux): FAIR8 — align MetadataCompleteness rubric with F-UJI`). The
old score rewarded having a hero image (5 pts). The new score rewards having
keyword annotations. The live LUMEN and MFFD Collections now have hero images
but score 0 on the keywords check. If the keyword endpoint shipped at the
same time as FAIR8, those Collections would gain 5 pts immediately because
they already carry semantic annotations on their DataObjects that could be
aggregated as keyword signals. The two commits are pushing in opposite
directions on a metric that matters to F-UJI auditors.

**AAS1l (config singleton) + AAS-CODE-EXTRACTION (queued) are subtly in
tension.** AAS1l landed V88 migration in backend resources
(`backend/src/main/resources/neo4j/migrations/V88__aas_config.cypher`) while
the plugin is `plugins/aas/`. This mirrors the V46 migration that was left
in-tree during AAS1-plugin extraction, which the backlog row `AAS-CODE-EXTRACTION`
explicitly notes as debt. Each AAS milestone that adds a migration to the
backend resources deepens the extraction surgery needed later. The pattern is
established but worth flagging: the next AAS migration should consider whether
it can ship as a plugin-declared migration (if Flyway callbacks support it)
rather than landing in the core migration chain.

**The brush-range time-bracket in the playground and the forthcoming Gantt
VIEW_RECIPE (`VIS-GANTT`) share a temporal-window selection problem.** The
playground uses `brushSlider` (0–100 percentage range → nanosecond timestamp
range). The Gantt will need a similar mechanism for zooming into a sub-segment
of a process timeline. Designing the Gantt's time-window widget to emit the
same `(startNs, endNs)` pair as the playground's brush range would mean
Trace3D and Gantt share a single temporal-selection primitive — a user could
select a time window in the Gantt and the Trace3D view would update to that
window. This composability follows directly from the VIEW_RECIPE design but is
not yet articulated in `VIS-GANTT`'s backlog row.
