## 1. What this repo is

`/opt/shepard` is a fork of `gitlab.com/dlr-shepard/shepard` (upstream tag 5.2.0). The upstream is a research data management platform from DLR (Deutsches Zentrum für Luft- und Raumfahrt). This fork adds:

- A new `/v2/` API surface (the upstream `/shepard/api/` surface is frozen for backwards compatibility)
- User profile features: avatar, ORCID auto-sync, display name, preferences
- Timeseries charting with live-mode
- Collection lineage graph and data-object provenance view
- Admin features (feature toggle registry, per-shelf OpenAPI, semantic ontology admin)
- CI pipelines (GHCR image builds, Kiota client codegen, security gates)

**The rule:** `/shepard/api/...` is byte-compatible with upstream 5.2.0 forever. All new endpoints go under `/v2/`.

**Infrastructure:** Docker Compose at `/opt/shepard/infrastructure/docker-compose.yml` (override at `docker-compose.override.yml`).  
**Proxy:** Caddy (`infrastructure/proxy/Caddyfile`) — handles TLS termination and routing.

**Rebuild flow:**
```bash
# Backend:
cd /opt/shepard/backend && ./mvnw package -DskipTests -Dmaven.test.skip=true
docker build -t ghcr.io/noheton/shepard-backend:latest backend/

# Frontend (must rebuild backend-client/dist/ first!):
cd /opt/shepard/backend-client && npm run build
docker build -t ghcr.io/noheton/shepard-frontend:latest frontend/

# Redeploy:
cd /opt/shepard/infrastructure && docker compose up -d --force-recreate [backend|frontend]
```

**Critical:** Always run `cd /opt/shepard/backend-client && npm run build` before building the frontend image. The frontend resolves `@dlr-shepard/backend-client` from `dist/` (gitignored), not from `src/`.

## 4. Key design decisions

### API versioning
- `/shepard/api/` — upstream frozen, never break. Bug fixes only.
- `/v2/` — all new endpoints. Uses `appId` (UUID) as primary identifier.

### Identity: appId vs OGM id
- Neo4j OGM assigns integer IDs. These are unstable across restores.
- `appId` is a stable UUID assigned on first `createOrUpdate` via `GenericDAO`.
- All `/v2/` endpoints use `appId`. Frontend passes both (the route param is the OGM integer for upstream compat; the v2 endpoints resolve via `EntityIdResolver`).

### Auth
- Keycloak (OIDC). JWTFilter validates token → UserFilter upserts the User node.
- Public paths: `PublicEndpointRegistry` — has both a `PUBLIC_PATH_PREFIXES` set (literal prefix match) and a `PUBLIC_PATH_PATTERNS` set (regex, used for `/v2/users/{appId}/avatar`).
- Backend client token: `useV2ShepardApi` reads `accessToken` from `useAuth()` (sidebase/nuxt-auth).

### Backend client double-encoding bug (fixed 2026-05-19)
`MeApi.ts` PATCH methods must pass the raw JS object to `BaseAPI.request()`, NOT pre-stringify it. `createFetchParams` in `runtime.ts` already serialises any `Content-Type: application/merge-patch+json` body. Pre-stringifying causes double-encoding → 400 from backend.

### CC1b "Referenced by" query (fixed 2026-05-19)
The relationship from `TimeseriesReference`/`StructuredDataReference`/`FileBundleReference` to their container uses `[:is_in_container]`, NOT `[:has_payload]`. The DAO queries were using `has_payload` (wrong edge) — all three DAOs now use `is_in_container`.

### Cache-control
Frontend HTML pages send `Cache-Control: no-store, must-revalidate` via Nuxt `routeRules`. Hashed `/_nuxt/` assets get `immutable` + 1 year. This is set in `nuxt.config.ts`, NOT in Caddy — Caddy only handles TLS and routing.

## 9. How to add a new /v2/ endpoint

1. Create the JAX-RS resource under `backend/src/main/java/de/dlr/shepard/v2/<feature>/resources/`.
2. Register the path in `PublicEndpointRegistry` if unauthenticated access is needed.
3. Add the endpoint to `backend-client/src/apis/` (hand-maintained until Kiota replaces it).
4. Run `cd backend-client && npm run build` before testing the frontend.
5. Update `aidocs/34-upstream-upgrade-path.md` with the new endpoint row.
6. Add a Playwright e2e test in `e2e/tests/`.
