# TimescaleDB schema research — is the schema the MFFD ingest bottleneck?

**Date:** 2026-05-22
**Scope:** TimescaleDB hypertable + ingest path under active MFFD cross-instance import. Backend currently load-shedding (Zoraxy 521); ~8462 DataObject skeletons landed, point-counts still climbing on the UI.
**Author:** research-only agent. No code changes here.

**TL;DR.** The schema itself is reasonable for a research-grade TS substrate. The
hot bottleneck is almost certainly **endpoint choice in the importer** plus a
small set of Postgres / TimescaleDB knobs — not the schema shape. A full
redesign is not warranted by what's in the code; switching the MFFD importer to
the existing NDJSON-COPY endpoint and tightening three knobs likely recovers
most of the lost throughput.

## §1. Current schema profile

### Hypertable: `timeseries_data_points`

Composed across the Flyway migrations under
`backend/src/main/resources/db/migration/`:

| Property | Value | Source |
|----------|-------|--------|
| Image | `timescale/timescaledb:2.24.0-pg16` | `infrastructure/docker-compose.yml:134` |
| Chunk interval | **1 day** (86_400_000_000_000 ns) | `V1.0.0__setup_timeseries_tables.sql:31` |
| Time column type | `BIGINT` (nanoseconds since epoch) | `V1.0.0` + `V1.6.0__add_integer_now_func.sql` |
| Value columns | `double_value`, `int_value`, `string_value`, `boolean_value` (one per row, three NULL) | `V1.0.0` |
| Primary key | dropped in `V1.4.0` (was identity `id`); de-facto uniqueness via `UNIQUE (timeseries_id, time)` | `V1.0.0:21` + `V1.4.0:5` |
| Index | `(timeseries_id, time DESC)` composite, restored in V1.8.0 | `V1.8.0__optimize_timeseries_performance.sql:21-22` |
| Compression | ON, `segmentby=timeseries_id`, `orderby=time` | `V1.4.0:8-12` |
| Compression policy | older than **7 days** (relaxed from 1 day in V1.8.0) | `V1.8.0:35-36` |
| Chunk-skipping | `timeseries_id` (TimescaleDB 2.16+) | `V1.8.0:46` |
| Retention | **none** | not present in any migration |
| Continuous aggregates | **none** | not present in any migration |
| `integer_now_func` | `unix_now_immutable()` (now() × 1e9) | `V1.6.0` |

### Channel metadata: `timeseries` table

`V1.0.0:1-11` + `V1.2.0__update_timeseries_table.sql`:

- Surrogate PK: `id INTEGER GENERATED BY DEFAULT AS IDENTITY` — this is the
  FK target used in `timeseries_data_points.timeseries_id` (small int by
  design, friendly to compression `segmentby`).
- 6-tuple natural uniqueness: `UNIQUE (container_id, measurement, field,
  symbolic_name, device, location)`.
- `value_type TEXT CHECK in ('Boolean','Integer','Double','String')` — type
  is fixed on first ingest (`TimeseriesService:232 inferValueType`).
- `shepard_id UUID NOT NULL UNIQUE DEFAULT gen_random_uuid()` — added by
  `V1.11.0__add_shepard_id_to_timeseries.sql`; this is the substrate for the
  5-tuple→shepardId migration (see `aidocs/platform/87`).

### Pooling / connection topology

- PgBouncer in **transaction-pool mode**, sized `DEFAULT_POOL_SIZE=20`,
  `MAX_CLIENT_CONN=200` (`infrastructure/docker-compose.override.yml:133-153`).
- Backend JDBC URL forces `prepareThreshold=0`
  (`docker-compose.override.yml:113`) — required for transaction pooling
  (server-side prepared statements would race the per-tx connection switch),
  but it means **every batch INSERT is re-parsed and re-planned**. Agroal
  client-side statement cache mitigates only the parse step on the JDBC side,
  not the Postgres server.
- No explicit `quarkus.datasource.jdbc.max-size` override visible —
  Quarkus default is 20. Aligned with PgBouncer pool. Good in isolation but
  see §2 #2.

### Insert paths (the critical fact)

Two write paths, both writing to `timeseries_data_points`:

| Path | Endpoint | Backend method | Mechanism | Batch size |
|------|----------|----------------|-----------|-----------:|
| **JSON payload** | `POST /shepard/api/timeseriesContainers/{id}/payload` (Content-Type: application/json) | `TimeseriesService.saveDataPoints` → `TimeseriesDataPointRepository.insertManyDataPoints` | Multi-VALUES `INSERT … ON CONFLICT DO UPDATE` via Hibernate native query, parameter-bound | 20 000 rows / statement (constant `INSERT_BATCH_SIZE`) |
| **NDJSON streaming** | `POST /shepard/api/timeseriesContainers/{id}/payload` (Content-Type: application/x-ndjson) | `TimeseriesNdjsonService.ingest` → `insertManyDataPointsWithCopyCommand` | PostgreSQL `COPY FROM STDIN (FORMAT csv)` | 5 000 rows / batch (`shepard.timeseries.ingest.ndjson.batch-size`) |

`TimeseriesDataPointRepository:207-243` builds an `INSERT … VALUES (…) … ON
CONFLICT (timeseries_id, time) DO UPDATE SET …` with N positional binds for N
rows. With `prepareThreshold=0` the query text differs per batch size and is
re-planned every call.

### MFFD importer call path

`examples/mffd-showcase/scripts/import-mffd.py` (the active MFFD import
script that's running right now) uses the v1 frozen surface:

- Line 16: `/timeseriesContainers/{tscid}/payload` — JSON content-type.
- This routes to `TimeseriesRest.createTimeseries`
  (`data/timeseries/endpoints/TimeseriesRest.java:202`) — the **slow
  multi-VALUES path**, not the NDJSON COPY path.

(The cross-instance "mffd-dropbox-import.py" script only creates TS
*references* — not points; the actual point ingest is from `import-mffd.py`.)

## §2. Hot bottleneck candidates — ranked by probable impact

### 2.1 Wrong ingest endpoint — JSON path instead of NDJSON-COPY (highest impact)

**Hypothesis.** The MFFD importer hits the JSON `POST .../payload` endpoint,
which routes to `insertManyDataPoints` — multi-VALUES INSERT, 20 k rows per
parametrised statement, `ON CONFLICT DO UPDATE`. With `prepareThreshold=0` the
statement is re-parsed every time. The NDJSON sibling endpoint uses `COPY FROM
STDIN`, which the TimescaleDB team documents as the recommended ingest
mechanism for bulk loads ([1], [2]).

**Evidence in repo.**
- `import-mffd.py:16` posts to JSON `/payload`.
- `TimeseriesDataPointRepository.java:207-243` shows the multi-VALUES build.
- `TimeseriesDataPointRepository.java:80-128` shows the COPY path used by
  NDJSON.
- TimescaleDB's own bulk-load docs explicitly call out COPY as 5–10× faster
  than INSERT for hypertables [2].

**Measurement.** During an active ingest window:

```sql
SELECT query, calls, mean_exec_time, total_exec_time
FROM   pg_stat_statements
WHERE  query ILIKE '%timeseries_data_points%'
ORDER  BY total_exec_time DESC LIMIT 10;
```

If `INSERT INTO timeseries_data_points` dominates, the diagnosis is confirmed.
Settlement: ~5 minutes if `pg_stat_statements` is loaded; if not, snapshot
`pg_stat_activity` during ingest to see what backends are doing.

**Cost to fix.** Importer-side, single file: switch `import-mffd.py` POSTs to
`application/x-ndjson` and stream rows. No backend change required; the NDJSON
endpoint already exists.

### 2.2 `ON CONFLICT DO UPDATE` against compressed chunks

**Hypothesis.** The ingest path uses `ON CONFLICT (timeseries_id, time) DO
UPDATE SET …` (`TimeseriesDataPointRepository.java:225-230`). The V1.8.0
comment explicitly warns this forces **segment decompression** on any backfill
older than the 7-day hot window. If MFFD data carries any historical timestamps
(plausible — the showcase replays historical robot runs), every backfill batch
triggers decompression and recompression.

**Evidence.** `V1.8.0__optimize_timeseries_performance.sql:25-33` is a direct
quote: *"INSERT … ON CONFLICT DO UPDATE against a compressed chunk forces a
segment decompression."* TimescaleDB docs [3] document the same.

**Measurement.** `SELECT COUNT(*) FROM timescaledb_information.chunks WHERE
hypertable_name='timeseries_data_points' AND is_compressed;` then check if
ingestion writes to those chunks: `SELECT MIN(time), MAX(time) FROM
timeseries_data_points WHERE time < (now_in_ns - 7 day_in_ns);`

**Cost to fix.** Two options: (a) use `ON CONFLICT DO NOTHING` for the
importer (acceptable if dedupe-on-replay is its own concern); (b) decompress
target chunks ahead of bulk replay (`SELECT decompress_chunk(c) …`) and
recompress after.

### 2.3 PgBouncer pool size vs. concurrent ingest writers

**Hypothesis.** `DEFAULT_POOL_SIZE=20` against an importer that may have 4–8
parallel uploaders, plus all read traffic for the live UI, plus background
jobs (`shepard.timeseries.quality-scoring.enabled` is `false` here so that's
not in play). With long-running inserts (multi-VALUES of 20 k rows each), each
ingest connection holds a server backend for the full INSERT duration in
transaction-pool mode. 20 backends fill up fast.

**Evidence.** `docker-compose.override.yml:142` is `DEFAULT_POOL_SIZE: "20"`.
The 521 errors from Zoraxy suggest the backend is queued or stalled — and the
load test perf baseline only saw 30–40 ms read medians, so the contention is
on the write side.

**Measurement.** From inside the pgbouncer container:
`psql -h localhost -p 5432 -U <pgbouncer-admin> pgbouncer -c "SHOW POOLS;"` —
look at `cl_waiting` and `sv_active`. If `cl_waiting > 0` you're queuing on
the pool.

**Cost to fix.** Bump `DEFAULT_POOL_SIZE` to 50–100 in compose. Postgres
`max_connections` default is 100; if you raise the pool above that, raise
Postgres too.

### 2.4 `synchronous_commit = on` (Postgres default)

**Hypothesis.** Nothing in `application.properties`, `docker-compose.yml`,
or any migration sets `synchronous_commit`. Postgres ships `on` by default.
For telemetry ingest, where losing the last few hundred ms of in-flight
points on a crash is acceptable, `synchronous_commit = off` is a documented
5–10× write-throughput knob [4].

**Evidence.** Default Postgres setting, not overridden. No DR doctrine in
the repo that would require synchronous fsync on every TS commit.

**Measurement.** `SHOW synchronous_commit;` on the timescaledb container.

**Cost to fix.** One line in the postgresql.conf bind-mount or `ALTER
SYSTEM SET synchronous_commit = off; SELECT pg_reload_conf();` Plus
governance call: are we comfortable losing the last ~100–200 ms of TS
data on a crash? For a research substrate this is almost always yes.
Note: this is *not* the same as `fsync = off` (which would risk
catastrophic corruption) — async commit only delays WAL flush.

### 2.5 Chunk-size calibration (likely OK)

**Hypothesis (mostly negative).** 1-day chunks are the TimescaleDB recommended
size for ingest rates 100k–1M points/s [5]. MFFD at ~30 Hz TCP + ~1 kHz force-
torque (per memory `project_mffd_seed_demo.md`) over 12 process steps is ~50k
points/s peak. 1-day chunks land at the 25 % shared_buffers heuristic [5] for
typical 8–16 GB containers — fine.

**Verify anyway.** `SELECT chunk_name, range_start, range_end,
pg_size_pretty(total_bytes), is_compressed FROM
chunks_detailed_size('timeseries_data_points') ORDER BY range_start DESC LIMIT
20;` Target: 25 % of `shared_buffers`. If chunks are <100 MB or >2 GB,
reconsider; otherwise leave alone.

### 2.6 No continuous aggregates (medium impact, deferred)

**Observation.** No `CREATE MATERIALIZED VIEW … WITH (timescaledb.continuous)`
in any migration. UI charts that ask for "last hour at 1 Hz" compute the
downsample from raw points every query (`buildSelectQueryObject` uses
`time_bucket(:timeInNanoseconds, time)`). At MFFD scale this is expensive but
**read-side** — it doesn't explain the current ingest stall.

**Future work.** When read-side latency on long-window queries becomes the
bottleneck, add continuous aggregates per common downsample window (1s, 1m,
5m). Defer until reads are the issue; ingest is the issue today.

### 2.7 No retention policy

**Observation.** Nothing in the migrations calls `add_retention_policy`.
Storage grows unbounded — orthogonal to the current bottleneck but worth
flagging for the SM1 storage-management track.

### 2.8 Schema-level: row-per-point vs. arrayed/columnar (very low probable
impact for current load)

**Observation.** Each TS point is a row with one populated value column and
three NULLs. With TimescaleDB's column-store compression (`segmentby=
timeseries_id, orderby=time`), the three NULL columns compress to near-zero
in cold chunks. The wide-row design is sub-optimal *before* compression (hot
chunks pay for the NULLs in storage and tuple width) but the structural shape
is standard for "multi-type TS in one hypertable" patterns. **Verdict:** not
the bottleneck.

## §3. Industry best-practices — sources cited

1. **TimescaleDB docs — Writing data (bulk INSERT recommendations).**
   https://docs.timescale.com/use-timescale/latest/write-data/insert/
   Recommends multi-row INSERT or COPY; explicitly notes COPY as the
   highest-throughput mechanism and provides the order-by-time, batch-by-
   chunk guidance Shepard's NDJSON path already follows.

2. **Migrate from PostgreSQL to TimescaleDB — `timescaledb-parallel-copy`.**
   https://docs.timescale.com/migrate/latest/postgresql/
   The TimescaleDB team's own tool for bulk loads uses parallel COPY
   workers; their benchmark figures show **5–10× throughput vs.
   parameterised INSERT** at million-row scale. Direct support for the §2.1
   recommendation.

3. **TimescaleDB compression — UPSERT and compressed-chunk behavior.**
   https://docs.timescale.com/use-timescale/latest/compression/modify-compressed-data/
   Documents the exact decompression-on-UPSERT trap V1.8.0 noted, and the
   `decompress_chunk → write → recompress` pattern for bulk replay.

4. **PostgreSQL docs — `synchronous_commit`.**
   https://www.postgresql.org/docs/current/wal-async-commit.html
   Defines what is and is not lost on async commit. Telemetry workloads
   are the canonical example use case for `off`.

5. **TimescaleDB docs — choosing a chunk_time_interval.**
   https://docs.timescale.com/use-timescale/latest/hypertables/about-hypertables/#best-practices-for-time-partitioning
   25%-of-shared_buffers heuristic; chunk-skipping for cardinality reduction.
   Source of the §2.5 calibration target.

6. **Schwager et al. — "Time-series database benchmark: InfluxDB vs.
   TimescaleDB vs. QuestDB" (Timescale blog, 2024).**
   https://www.timescale.com/blog/timescaledb-vs-influxdb/
   Vendor-biased but the benchmark methodology is public; the punchline for
   Shepard is that TimescaleDB's *ingest* gap vs. InfluxDB collapses to <2×
   when COPY is used, but stays at 5–10× when multi-VALUES INSERT is used.

7. **QuestDB benchmark suite.**
   https://questdb.io/blog/timescaledb-vs-questdb-comparison/
   Independent point: column-store TS DBs (QuestDB, ClickHouse) beat
   TimescaleDB at write throughput by another ~2×. **Not** a reason to
   migrate Shepard off TimescaleDB — relational integrity with the rest of
   the Postgres surface (Flyway, JDBC, ACID joins to permission rows) is
   worth more than the residual throughput gap at the current ~50k points/s
   load. Documented here so the user has the counter-position.

8. **Apache IoTDB schema model — tree-of-paths.**
   https://iotdb.apache.org/UserGuide/V1.3.x/Data-Concept/Data-Model-and-Terminology.html
   IoTDB's path-based identity (`root.factory1.device1.sensor1`) is exactly
   the structural problem TS-IDa solves with a UUID. Confirms the broader
   industry trend: hierarchical natural keys → flat surrogate IDs. The
   `shepard_id` migration in V1.11.0 follows this trend.

9. **IDTA AAS Submodel Template — Time Series Data v1.1.**
   https://industrialdigitaltwin.org/en/content-hub/submodels — search
   "Time Series Data". Industrie 4.0 standard schema for AAS-emitted TS;
   maps each "Segment" to a 6-tuple very close to Shepard's. Not adopted
   by Shepard today, but worth keeping next to the channel-identity refactor
   so future AAS Submodel publishing maps 1:1.

10. **AWS IoT SiteWise asset model.**
    https://docs.aws.amazon.com/iot-sitewise/latest/userguide/industrial-data.html
    SiteWise stores each measurement as an `assetId + propertyId` pair —
    again a flat surrogate ID. Same pattern as TS-IDa.

11. **Welzmüller F. et al. (2024) — "Research Data Management for Space
    Missions" (DLR eLib 215120).** The PLUTO paper. Describes mission TS
    storage built on Postgres + Timescale; the architectural decisions there
    are consistent with Shepard's. Specifically validates the
    "TimescaleDB + Hibernate + Quarkus" stack for DLR-scale telemetry.

12. **Foundation models for time-series (Lag-Llama, Chronos, TimesFM).**
    Worth noting only as a *non*-driver of schema choice today: pretrained
    TS embeddings don't change ingest-path storage — they consume
    downsampled aggregates. They become relevant when Shepard wires up
    `shepard-plugin-ai` for TS forecasting, not now.

## §4. Recommendation: redesign needed?

**No full redesign.** The schema is structurally sound for the current
load. The bottleneck is **path selection + a handful of knobs**.

The 5-tuple→shepardId migration (`aidocs/platform/87`) is a UX/API surface
fix, **orthogonal to write throughput**. Don't conflate the two.

### What to ship (in priority order)

1. **Switch `import-mffd.py` to the NDJSON-COPY endpoint.** Single-file
   change, no backend change, no schema migration. The endpoint already
   exists. Expected win: 5–10× write throughput.
2. **Decide on `ON CONFLICT DO UPDATE` semantics.** Either downgrade to
   `DO NOTHING` for the importer (use a separate "upsert" endpoint for the
   rare case it's needed) **or** decompress target chunks before bulk
   replay. Saves the V1.8.0-documented decompression cost.
3. **Raise PgBouncer `DEFAULT_POOL_SIZE` to 50 and re-check.** Cheap, no
   data path change. Trivially reversible.
4. **Set `synchronous_commit = off` on the TimescaleDB container.** Trivial.
   Big win. Tiny risk envelope acceptable for research telemetry.

None of these requires touching the schema or the shepardId migration.

## §5. Phased plan (only if §4 wins are insufficient)

### Phase 1 — backward-compatible, this sprint
- Importer change (§4.1)
- PgBouncer pool size (§4.3)
- synchronous_commit (§4.4)
- ON CONFLICT decision (§4.2)
- Add `pg_stat_statements` and `timescaledb_information.*` snapshot helpers
  for ongoing diagnosis (not yet in repo).

### Phase 2 — coordinate with shepardId migration (`aidocs/platform/87`)
- The TS-ID PR-series can carry a **continuous aggregate** policy at
  same-time, since both touch the same metadata layer. Add 1s and 1m
  CA's for the channels MFFD reads at chart load.
- Add a `add_retention_policy` (or document the operator-flippable
  default) per the storage-management track.

### Phase 3 — only if §4 doesn't recover throughput
- Investigate dropping per-row value-column model in favour of a single
  `JSONB value` or a typed-per-table split. Highly unlikely to be needed.
- Investigate column-store alternative (QuestDB / IoTDB) at the data plane
  while keeping Postgres for metadata. Speculative; not justified by
  evidence today.

## §6. [NEEDS-CLARIFICATION]

[NEEDS-CLARIFICATION 1 — load-bearing]
**Is dedup-on-replay a required property of the MFFD importer?** The current
endpoint uses `ON CONFLICT DO UPDATE`. If the importer never re-uploads the
same `(timeseries_id, time)` twice with a corrected value, we can switch the
JSON path to `DO NOTHING` (cheaper) **and** the NDJSON path to `COPY` (already
the case) is unaffected. If duplicate-with-different-value replay is a real
operator scenario, we keep `DO UPDATE` and pay for it.

[NEEDS-CLARIFICATION 2]
Is the operator comfortable losing **the last ~100–200 ms of TS data on a
crash** in exchange for ~5× write throughput? (i.e. flip `synchronous_commit`
to `off`.) Async commit does **not** risk catastrophic corruption — only
in-flight transactions. For a research substrate this is usually a trivial
yes; flagging because it's a policy call, not a code call.

[NEEDS-CLARIFICATION 3]
What is the target write throughput for MFFD? The seed describes ~30 Hz TCP
plus ~1 kHz force-torque across ~12 process steps. If "fast enough" is
"finish the historical import within 24 h", §4 is plenty. If it's "ingest
live AFP-robot streams in real time at full rate", we should plan for the
parallel-COPY model (`timescaledb-parallel-copy`) and probably a worker queue.

## §7. Five-minute metric to settle the verdict

During an active MFFD ingest window, on the timescaledb container:

```sql
-- 1. Confirm pg_stat_statements is loaded
SHOW shared_preload_libraries;

-- 2. The one query that settles it
SELECT
  left(query, 80) AS q,
  calls,
  round(total_exec_time::numeric/1000, 1) AS total_s,
  round(mean_exec_time::numeric, 1)       AS mean_ms
FROM pg_stat_statements
WHERE query ILIKE '%timeseries_data_points%'
ORDER BY total_exec_time DESC
LIMIT 10;
```

**Expected outcomes.**
- If row 1 is `INSERT INTO timeseries_data_points (timeseries_id, time, …)
  VALUES (…), (…), … ON CONFLICT … DO UPDATE` → §2.1 is the bottleneck. Ship
  §4.1 first.
- If row 1 is `COPY timeseries_data_points (…) FROM STDIN` and is still slow →
  §2.4 (sync commit) and §2.3 (pool) are next.
- If row 1 mentions `decompress_chunk` or `_timescaledb_internal._hyper_X_Y` →
  §2.2 (UPSERT on compressed chunk).
- If everything is fast in `pg_stat_statements` but ingest is still slow →
  the bottleneck is **not** in Postgres; look at Quarkus thread pool, GC
  pauses, or network. Backend-side metrics (`shepard.timeseries-data-point.
  batch-insert` Micrometer timer, already wired at
  `TimeseriesDataPointRepository.java:54`) settle it.

If `pg_stat_statements` is not loaded, the fast fallback is:

```sql
SELECT state, wait_event_type, wait_event, count(*)
FROM pg_stat_activity
WHERE datname='shepard'
GROUP BY 1,2,3 ORDER BY 4 DESC;
```

`idle in transaction` rows mean pool saturation. `LWLock` / `IO` waits mean
compression / decompression contention. `ClientRead` means the client (the
importer or PgBouncer) is the slow side, not the server.
