# In-App User Docs — Design

**Scope.** Make user documentation **directly accessible inside
the shepard UI** — a casual user stuck on a task gets help without
leaving the app. Screenshots in those docs are generated by
Playwright against the live test deployment
(`shepard.nuclide.systems` per `aidocs/40 §3c`) and committed
back to the repo, so the docs always reflect what the user is
actually looking at.

**Status.** Concept design.
**Snapshot date.** 2026-05-08.
**Originating items.** User request: "shepard UI should contain
directly hosted user docs. we generate the necessary screenshots
when the test instance is available." Couples to:
`aidocs/33-frontend-workflow-analysis.md` (already names Playwright
as the screenshot tool for R5 / W2-W11), `aidocs/42-vision.md` §1.0
casual-user north star, `docs/` Pages site (existing user docs
source), `docs/showcase.md` (already uses screenshot placeholders).

---

## 1. Why

The current state is an **off-domain** docs site at
`https://noheton.github.io/shepard/` (`docs/` built by Jekyll +
GitHub Pages). It's good — but a casual user doesn't think to
leave shepard's tab to go find it. Three concrete failure modes:

1. **Context switch tax.** "Where's the export button?" → user
   alt-tabs to a search engine → finds Stack Overflow or a wiki
   that's outdated → comes back confused. **In-app help would
   answer in two clicks.**
2. **Version drift.** Pages-site screenshots pinned to whatever
   the maintainer happened to capture last. If the shepard build
   the user runs has a different UI, the screenshots are wrong
   and trust erodes silently.
3. **No screenshot story today.** `docs/showcase.md` has
   `<figure class="screenshot-placeholder">` markers waiting for
   captures to land. They've been waiting because there was no
   pipeline. This design ships the pipeline.

The **vision link**: `aidocs/42 §1.0` says "casual users come
first." If the casual user can't find the docs, the docs don't
exist.

## 2. Architecture

Two surfaces serve the same content:

```
              docs/ (Markdown source)
                    │
                    ▼
       Jekyll build (existing — pages.yml)
                    │
        ┌───────────┴────────────┐
        ▼                        ▼
GitHub Pages site         In-app /help bundle
(public, off-domain)      (served by shepard
                           backend or the
                           frontend assets)
```

One source of truth (`docs/*.md`), two presentations. Critically:
**screenshots live in `docs/assets/screenshots/` and both surfaces
consume them**. The Playwright pipeline (§4) generates that
directory; a missing screenshot shows a placeholder in both
surfaces.

### 2.1 In-app rendering — the frontend route

A new `/help` route in the Nuxt 3 frontend. Two implementation
shapes considered:

| # | Shape | Verdict |
|---|---|---|
| (i) | **Iframe pointing at the live Pages site** | Rejected. Air-gapped / on-prem deployments can't reach `noheton.github.io`. CSP / cookie boundaries break in-app navigation polish. |
| (ii) | **Bundle the rendered HTML into the frontend assets at build time; serve from same origin** | **Recommended.** Self-contained; works offline; same origin so navigation feels native. Increases the frontend bundle by the size of `docs/_site/` (~1-2 MB compressed without screenshots, ~5-15 MB with full screenshot set). |

Path (ii) shape:

- Frontend build step copies `docs/_site/` → `frontend/public/help/`.
- A small Vue component (`HelpFrame.vue`) renders the static HTML
  inline using `v-html` against trusted bundled content. Or just
  uses an `<iframe src="/help/{topic}">` for sandbox isolation —
  whichever the existing frontend's link/asset patterns prefer.
- Top nav gets a "Help" item linking to `/help`; same pane shape
  as `aidocs/33 §W12` Configuration.

### 2.2 Doc-source layout — two-track structure

`docs/` stays the source. The user-facing content is organised in
**two tracks under one nav** so casual users land on a friendly
front door but power users still find every feature documented:

**Track A — "Get started / Common tasks" (casual landing).**
Task-shaped, ≤ 5-minute path-to-success per page, heavy on
screenshots, links forward to the reference for depth. Lives at
`docs/help/`:

- `docs/help/index.md` — the `/help` landing page. "I want to do
  X" picker linking into the task pages.
- `docs/help/upload-data.md` — "How do I get my data into shepard?"
- `docs/help/share-collection.md` — "How do I share with a
  colleague?"
- `docs/help/export-rocrate.md` — "How do I export for
  publication?"
- `docs/help/process-step.md` — "How do I follow a templated
  process?"
- `docs/help/find-data.md` — "How do I find a dataset I (or someone
  else) created?"
- `docs/help/annotate.md` — "How do I tag data with an ontology
  term?"
- `docs/help/lab-journal.md` — "How do I write a lab journal entry?"
- `docs/help/dashboard.md` — "How do I get a chart from my data?"
  (post-AI1e snap dashboards)

**Track B — "Reference / Every feature" (comprehensive).**
Feature-shaped, complete per shipped feature. Lives at the existing
`docs/` top-level + a new `docs/reference/` for per-primitive
deep-dives:

| Primitive | Reference page |
|---|---|
| Collection | `docs/reference/collection.md` |
| DataObject | `docs/reference/data-object.md` |
| FileReference | `docs/reference/file-reference.md` |
| StructuredDataReference | `docs/reference/structured-data-reference.md` |
| TimeseriesReference | `docs/reference/timeseries-reference.md` |
| SpatialDataReference | `docs/reference/spatial-data-reference.md` |
| HdfReference *(post-A5)* | `docs/reference/hdf-reference.md` |
| GitReference *(post-G1)* | `docs/reference/git-reference.md` |
| Permissions | `docs/reference/permissions.md` |
| Annotations | `docs/reference/annotations.md` |
| Lab journal | `docs/reference/lab-journal.md` |
| Versions / Snapshots *(post-V2)* | `docs/reference/snapshots.md` |
| Templates *(post-T1)* | `docs/reference/templates.md` |
| Processes *(post-PR1)* | `docs/reference/processes.md` |
| API keys + ORCID + profile *(post-U1)* | `docs/reference/profile.md` |
| AI features *(post-AI1)* | `docs/reference/ai-features.md` |
| Configuration / preferences | `docs/reference/preferences.md` |

Existing `user-guide.md` becomes the **bridge index** that links
into both tracks; `getting-started.md` stays the API-quickstart
flavour for power users; `architecture.md` / `system-requirements.md`
/ `admin.md` / `deploy*.md` stay where they are (admin/operator
audience, untouched).

**Track-A pages always link to track-B for depth.** Track-B
pages always link back to track-A "common tasks." A casual user
who needs more never hits a dead end; a power user who needs the
quick path doesn't have to scroll through reference prose.

### 2.2a The completeness standing rule

**Every user-visible feature must ship a user-doc page in the same
PR.** Tracked as a new standing rule in `CLAUDE.md` (§docs
currency); reviewers reject feature-shipping PRs without the
matching `docs/reference/*.md` (and a `docs/help/*.md` if the
feature has a casual-task expression). This is the structural fix
for "the docs always reflect what shepard does today" — same shape
as the vision (`aidocs/42`) and matrix (`aidocs/44`) currency
rules already in `CLAUDE.md`.

The catalogue in §2.2 above is the **target state** at full
feature parity. As of the snapshot date most reference pages are
**TBD** — they land per-feature as their slices ship. The
casual-landing pages (track A) ship at D1c as a unit so the
front-door feels complete from day 1 even when the reference
backbone is still filling in.

### 2.3 Versioning

Each shepard release pins the `docs/` content at that release. The
frontend's `/help` shows **the docs that match the running
backend version** — never a future version's docs against an older
backend.

Mechanism: the Jekyll build emits a `docs/_site/version.json` with
the backend's version string (read at build time from
`backend/pom.xml`'s `<revision>`). The frontend reads it and shows
"Help for shepard 5.2.0+noheton.4" or similar at the bottom of
every page. If a user really wants the latest docs (e.g. for a
forthcoming feature), they go to the public Pages site.

## 3. The Playwright screenshot pipeline

Nine months of `<figure class="screenshot-placeholder">` markers
in `docs/showcase.md` proves the pipeline matters more than the
docs do. Specs:

### 3.1 Where screenshots come from

A **Playwright spec** at `frontend/tests/e2e/screenshot-pipeline/`
runs against the **live test deployment**
(`https://shepard.nuclide.systems` per `aidocs/40 §3c` —
maintainer's existing self-hosted-behind-Zoraxy deployment). The
spec:

1. Authenticates with a dedicated read-only API key (kept in CI
   secret `SHEPARD_TEST_INSTANCE_APIKEY`).
2. Navigates the LUMEN showcase Collection (loaded via the seed
   per `examples/seed-showcase/seed.py`).
3. Captures one screenshot per `data-target="<id>"` marker in the
   `docs/` Markdown source.
4. Saves PNGs to `docs/assets/screenshots/<id>.png`.

The marker convention is already in place — see `docs/showcase.md`:

```html
<figure class="screenshot-placeholder" data-target="stop-01-collection-tree">
  <figcaption>Screenshot: stop 01 — Collection tree...</figcaption>
</figure>
```

The spec reads every `.md` under `docs/`, finds every
`data-target`, and ensures one PNG per marker.

### 3.2 When the pipeline runs

| Trigger | Reason |
|---|---|
| Manual via `workflow_dispatch` | Operator-initiated capture (e.g. after a UI change). |
| Weekly schedule | Catches gradual drift between the docs and the deployed UI. |
| On `main` after `Deploy to test instance` succeeds (`workflow_run`) | The deploy workflow (per `docs/deploy-self-hosted-zoraxy.md §5a.10`) ships the new build to `shepard.nuclide.systems`; the screenshot pipeline runs against the **freshly-deployed** UI. Catches per-PR drift end-to-end without waiting for the weekly. |
| On `main` after a merge that touches **`docs/help/*`** | Closed-loop for content-only changes that don't trigger an image rebuild. |

**Why chain off `Deploy to test instance`** (vs. running on every
`main` push directly): the deploy can take 1-2 minutes between
push and a healthy backend. Capturing screenshots against an
in-flight deploy would catch a 502 page mid-restart. The
`workflow_run` chain waits for the deploy's `Healthy after Nx2s`
smoke-test before the screenshot job starts.

### 3.3 Where screenshots live

`docs/assets/screenshots/<id>.png` — committed back to the repo
by the workflow. Why commit (vs. generate-on-build):

- **Reproducibility.** The Pages site builds offline; in-app
  bundle builds offline. No network dependency on a live
  deployment at build time.
- **History.** Git diff on a PNG isn't pretty, but it shows
  *that* the screenshot moved. Useful for "did the UI change
  between v5.2.0 and v5.3.0?"
- **Air-gap.** Operators on disconnected networks pull the repo
  and get the docs + screenshots in one shot.

The PNG repo footprint is bounded by the screenshot count (~30-50
expected, ~150 KB each → ~5-7 MB total). Git LFS not needed at
that scale.

### 3.4 The workflow file

New `.github/workflows/screenshot-pipeline.yml`:

```yaml
name: Screenshot pipeline
on:
  schedule:
    - cron: "13 4 * * 1"      # Mondays 04:13 UTC
  workflow_dispatch:
  push:
    branches: [ main ]
    paths:
      - "frontend/**"
      - "docs/**.md"
      - ".github/workflows/screenshot-pipeline.yml"

permissions:
  contents: write              # to commit the captured PNGs

jobs:
  capture:
    runs-on: ubuntu-latest
    env:
      SHEPARD_BASE: https://shepard.nuclide.systems
      SHEPARD_APIKEY: ${{ secrets.SHEPARD_TEST_INSTANCE_APIKEY }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'npm', cache-dependency-path: frontend/package-lock.json }
      - name: Install Playwright
        run: cd frontend && npm ci && npx playwright install --with-deps chromium
      - name: Capture screenshots
        run: cd frontend && npx playwright test tests/e2e/screenshot-pipeline/
      - name: Commit captured PNGs (if any changed)
        run: |
          git config user.name "shepard-bot"
          git config user.email "shepard-bot@users.noreply.github.com"
          git add docs/assets/screenshots/
          if ! git diff --cached --quiet; then
            git commit -m "docs: refresh in-app screenshots from shepard.nuclide.systems"
            git push
          fi
```

### 3.5 The Playwright spec shape

```typescript
// frontend/tests/e2e/screenshot-pipeline/showcase.spec.ts
import { test } from '@playwright/test';
import { readdirSync, readFileSync, mkdirSync } from 'fs';
import path from 'path';

const DOCS_DIR = path.resolve(__dirname, '../../../../docs');
const OUT_DIR  = path.resolve(__dirname, '../../../../docs/assets/screenshots');

const markers = collectMarkers(DOCS_DIR);  // walks *.md, finds data-target=

mkdirSync(OUT_DIR, { recursive: true });

for (const marker of markers) {
  test(`screenshot: ${marker.id}`, async ({ page }) => {
    await page.goto(marker.url);              // marker maps id → URL pattern
    await page.waitForLoadState('networkidle');
    await page.screenshot({
      path: path.join(OUT_DIR, `${marker.id}.png`),
      fullPage: false,
      clip: marker.clip,                      // optional per-marker viewport
    });
  });
}
```

The marker → URL mapping is defined in
`frontend/tests/e2e/screenshot-pipeline/marker-routes.ts`. New
markers in `docs/*.md` need a row in that file; absent a row, the
spec emits a placeholder PNG and a warning so the gap is visible
in the workflow output.

### 3.6 What if the test instance is down

The Playwright spec uses a `--retries=2` policy. If
`shepard.nuclide.systems` is unreachable, the workflow fails;
shepard-bot doesn't commit anything; existing screenshots stay
in place. The workflow run shows red so the maintainer notices.

For local-only operators who don't have a public test instance,
the spec also runs against `http://localhost:8080/shepard/` with
the LUMEN seed loaded — same fixtures, no public dependency.
Documented in `docs/admin.md`'s "Generating in-app docs" section.

## 6. Risks

- **PNG repo bloat over time.** ~5-7 MB at v1; grows with
  every new marker. Mitigation: Git LFS once we cross 50 MB; not
  needed at v1.
- **Test-instance-down → stale screenshots.** Workflow turns red;
  existing screenshots stay. Acceptable failure mode.
- **Test-instance compromise → poisoned screenshots committed.**
  Mitigation: the read-only API key is scoped to the LUMEN
  showcase Collection only; `shepard-bot` commits go to `main`
  via PR (§3.4 `git push` is a direct push — change to "open a PR
  with the diff" for tighter control if needed).
- **Frontend bundle size growth.** Bundle docs + screenshots
  inflates the JS bundle by ~5-15 MB. Mitigation: serve
  `frontend/public/help/` as **separate static assets**, not
  bundled into the JS chunk — only loaded when the user navigates
  to `/help`.
- **Screenshots show secrets.** A capture against the live test
  instance might catch an admin email, an API key fragment, etc.
  Mitigation: the Playwright spec runs as a read-only user, never
  visits admin / settings panes (only the showcase Collection +
  features specifically marked as docs-targets), and the
  marker-routes file is reviewed when new markers are added.
- **In-app docs and Pages site drift.** They share a source so
  drift is structural impossibility — but if D1d's version
  stamping isn't honoured rigorously, an old shepard could ship
  newer docs. Mitigation: build-time check that
  `docs/_site/version.json` matches `backend/pom.xml`'s
  `<revision>`.

---

## 7. Out of scope

- **Anonymous in-app docs.** A user not logged into shepard sees
  the public Pages site. The in-app `/help` route requires the
  user to be authenticated — same as every other shepard route.
- **Editable docs from the UI.** No "edit this page" button.
  Docs land via PRs to `docs/*.md`.
- **Search across docs.** Out of scope for D1; the AI snap
  dashboards (`aidocs/43 §5.1`) natural-language search is the
  forward path. v1 docs are flat-indexed via the page nav only.
