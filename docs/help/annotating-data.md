# Annotating data with semantic tags

Semantic annotations attach a **property–value** pair to any Shepard entity —
a data object, collection, file reference, timeseries container, or anything
else. Annotated entities become discoverable through search and queryable
through SPARQL.

---

## Annotate an entity (3-click flow)

1. Open the entity detail page (any data object, collection, or container).
2. Find the **Semantic Annotations** panel and click the **Annotate** button
   (or the **+** chip).
3. The **Annotate** dialog opens with two fields:
   - **Predicate** — what the annotation says about the entity (e.g.
     "description", "hasMeasurementPrinciple", "wasAttributedTo").
     Start typing to search across loaded vocabularies, or paste a full IRI.
   - **Value** — the thing being asserted (e.g. a concept from an ontology,
     an ORCID, or free text like "Turbopump vibration anomaly at t=8s").
     Again, type to search or paste an IRI. Free text is also accepted.
4. Click **Apply** (or press **Ctrl+Enter**).

A preview card appears under each field when you pick a known vocabulary
term, showing the term's label, description, and IRI so you can confirm
you have the right concept before saving.

---

## Optional fields

Click **Optional fields** in the dialog to reveal:

- **Source mode** — who produced this annotation:
  - 🧑 **Human** (default) — you typed it
  - 🤖 **AI** — generated by an AI model (set automatically when an
    AI calls the API with the `X-AI-Agent` header)
  - 🤝 **Collaborative** — AI suggested, human approved
- **Confidence** — a slider from 0.0 to 1.0. Human writes default to 1.0.
  Use lower values for uncertain or probabilistic assertions.
- **Valid from / Valid until** — ISO dates for time-bounded assertions.
  Leave blank for timeless statements.

---

## Find entities by annotation

Use the **Search** bar at the top of any collection, or open the
**Annotations** facet panel on the left. You can filter by:

- Predicate IRI (e.g. `dcterms:description`)
- Object value (free text)
- Source mode (`human` / `ai` / `collaborative`)

From the collection annotations panel, click any annotation chip to see all
entities in the collection carrying that term.

## Annotating many entities at once (batch via MCP)

For bulk annotation — for example, tagging all 200 data objects in a test
campaign with the same material specification — use the Shepard MCP tools
from Claude or another MCP-capable client:

```
create_annotation(
  subjectAppId="<the data object's appId>",
  subjectKind="DataObject",
  predicateIri="http://purl.org/dc/terms/description",
  objectLiteral="AFP layup — ply 5 consolidation pass"
)
```

You can ask Claude to loop over the entities returned by `list_data_objects`
and call `create_annotation` for each one. All annotations created this
way are recorded as `sourceMode: ai` with a provenance trail.

See [semantic annotations reference](/reference/semantic-annotations/#mcp-tools)
for the full list of available MCP tools.

## Annotations vs. attributes

| | Annotations | Attributes |
|---|---|---|
| Storage | Neo4j `:SemanticAnnotation` node | `attributes` map on `:DataObject` |
| Format | Structured IRI + optional IRI or literal | Free key-value strings |
| Queryable via SPARQL | Yes | No |
| MCP tools | Yes | Read-only (legacy) |
| Write | Always allowed for entities with Write permission | Write-disabled since SEMA-V6 |

Use **annotations** for all new metadata. The `attributes` map is a
read-only window onto pre-SEMA-V6 imports; it will not accept new writes.

---

## Permissions

- **Read** permission on a collection: see all annotations on entities
  in that collection.
- **Write** permission on a collection: create, update, and delete
  annotations on entities in that collection.

Annotations on public collections are visible to everyone, including
unauthenticated users.
