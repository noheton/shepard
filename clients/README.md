# `clients/` — OpenAPI Generator (legacy, still-maintained)

This directory is the **still-maintained, never-deprecated legacy** OpenAPI
Generator output for the byte-frozen `/shepard/api/...` surface, per
[`aidocs/63` ADR-0022](../aidocs/63-architecture-decision-log.md#adr-0022--openapi-client-generators-kiota-new-baseline--openapi-generator-still-maintained-legacy)
("OpenAPI client generators: Kiota new baseline + OpenAPI Generator
still-maintained legacy") and
[`aidocs/57 §4.2`](../aidocs/57-openapi-client-generator-evaluation.md#42-secondary-still-maintained-legacy-openapi-generator-for-shepardapi-continuity).

## TL;DR

- These clients cover **only** the upstream-compatible `/shepard/api/...`
  paths — the surface that this fork keeps byte-compatible with
  `gitlab.com/dlr-shepard/shepard 5.2.0` per `CLAUDE.md §API-version-policy`.
- They are generated by [OpenAPI Generator](https://openapi-generator.tech)
  (`openapitools/openapi-generator-cli`) from
  `/shepard/doc/openapi/v1.json` — the v1 shelf of the per-shelf split
  (P4c, shipped).
- The companion `clients-v2/` directory (CG1a, sibling work) carries the
  Kiota-generated SDKs for the fork's `/v2/...` development shelf.

## Choose your client

| You're calling… | Use… | Lives at |
|---|---|---|
| `/shepard/api/...` (upstream-compatible) | **OpenAPI Generator** clients | `clients/` (this directory) |
| `/v2/...` (this fork's development shelf) | **Kiota** clients | `clients-v2/` |
| Casual Python or TypeScript script | **`shepard-py` / `shepard-ts`** convenience wrapper (`aidocs/27`) | Separate repo (companion); wraps whichever client is best for the request |

Neither generator is on a deprecation path. Pick the one that fits your
endpoint surface; an operator using one today can keep using it
indefinitely.

## Per-language output

- **Java**: `clients/java/` — Maven artefact `de.dlr.shepard:shepard-client`
- **Python**: `clients/python/` — PyPI package `shepard-client`
- **TypeScript**: `clients/typescript/` — npm package `@dlr-shepard/shepard-client`

Each per-language sub-directory has its own README with the dual-posture
framing.

## Why we kept OpenAPI Generator instead of porting everything to Kiota

Three reasons, summarised from [`aidocs/57 §4.2`](../aidocs/57-openapi-client-generator-evaluation.md#42-secondary-still-maintained-legacy-openapi-generator-for-shepardapi-continuity):

1. **Zero-impact for upstream consumers.** Anyone built against
   `dlr-shepard-clients/*` keeps working unmodified — same generator,
   same templates, same wire shape. The fork's API-version policy
   (`CLAUDE.md`) makes this guarantee explicit at the wire-protocol
   level; CG1b extends it to the client-generation toolchain too.
2. **Best tool per surface.** OpenAPI Generator's mature templates are
   exactly what the upstream-frozen surface needs — no fluent
   path-builder noise, no MCP-extension surfacing (that's a `/v2/`
   feature). The clients here look the same as they always have.
3. **Two-shelf split makes it cheap.** The per-shelf OpenAPI emission
   (P4c, shipped) gives each generator its own canonical input —
   `/shepard/doc/openapi/v1.json` for this directory, `v2.json` for
   `clients-v2/`. No "OpenAPI Generator vs Kiota" rivalry to
   adjudicate — they target different paths.

## How generation works

1. The backend's `mvn package` emits `backend/target/openapi/openapi.json`
   (the combined OpenAPI document).
2. A build-time slicer (`scripts/shepard_scripts/scripts/slice_openapi_v1.py`)
   filters that down to `backend/target/openapi/openapi_v1.json` —
   mirroring the runtime `V1OpenApiFilter` (Java) one-for-one. The
   slicer drops `/v2/...` paths and platform paths (`/healthz`,
   `/openapi`, `/swagger-ui`, `/metrics`); everything else stays.
3. The CI pipeline ([`.gitlab/ci/clients/*.yml`](../.gitlab/ci/clients/))
   runs `openapitools/openapi-generator-cli` against
   `openapi_v1.json` per language and writes the result to
   `clients/<lang>/`.
4. A per-language **v1-scope smoke test** (`clients/tests/v1-scope-smoke-test.sh`)
   asserts:
   - the generated client contains classes / methods referencing
     `/shepard/api/...` endpoints (at minimum a "Collection"-shaped
     class — the upstream Collection API has been stable since 5.0.x),
     **and**
   - the generated client contains **no** `/v2/` references at all.

   This is the regression fence that locks the v1-shelf scope. If a
   future change accidentally retargets the input back to the combined
   `openapi.json`, the smoke test fails loudly in CI.

## Generating locally

See [`clients/tests/python/README.md`](tests/python/README.md) and
[`clients/tests/typescript/README.md`](tests/typescript/README.md) for
the language-specific recipes. Each starts by running the slicer to
produce `openapi_v1.json` and then invokes the generator against it.

## Operator upgrade impact

**None.** The retarget is behaviour-preserving for downstream
`dlr-shepard-clients` consumers — the generated client surface is the
same one upstream has been shipping, just generated from a v1-only input
spec instead of a combined spec with `/v2/` paths layered in. See
[`aidocs/34`](../aidocs/34-upstream-upgrade-path.md) for the
admin-facing tracker entry.
